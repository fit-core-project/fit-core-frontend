# Fit-Core AI Muscle Selection Normalization v1

작성일: 2026-04-18  
대상: Planner, Frontend, Backend, AI  
목적: 2D 근육 선택 UI에서 나오는 raw 값과 Fit-Core 내부 도메인/AI 입력값을 분리해, `react-body-highlighter` 같은 외부 라이브러리 변경이 저장 모델과 AI 계약을 직접 흔들지 않도록 한다.

## 1. 왜 이 문서가 필요한가

현재 팀 내에서 아래 세 가지가 섞여 있었다.

1. `react-body-highlighter`가 다루는 body part slug
2. 프론트 또는 기존 코드에서 쓰는 세부 코드 (`CHEST_UPPER` 등)
3. mock / catalog / AI prompt에서 쓰는 비교적 거친 근육 이름 (`chest`, `triceps` 등)

이 상태로 가면 아래 문제가 반복된다.

- 프론트 라이브러리 교체 시 백엔드/AI 계약이 흔들린다
- mock 데이터와 실제 UI 선택값이 계속 어긋난다
- DB / 계약 / AI 프롬프트가 같은 근육을 서로 다른 이름으로 부른다

따라서 이번 결정은 간단하다.

`외부 UI raw 값`과 `도메인 기준값`과 `현재 AI 입력값`을 분리한다.

## 2. 공식 라이브러리 기준에서 확인한 사실

`react-body-highlighter` README 기준으로:

- 예시 데이터는 `chest`, `triceps`, `front-deltoids` 같은 slug를 사용한다.
- 클릭 콜백은 `{ muscle: 'name', stats: ... }` 구조를 넘긴다.
- README에 적힌 지원 부위는 `trapezius`, `upper-back`, `lower-back`, `chest`, `biceps`, `triceps`, `forearm`, `back-deltoids`, `front-deltoids`, `abs`, `obliques`, `adductor`, `hamstring`, `quadriceps`, `abductors`, `calves`, `gluteal`, `head`, `neck`다.

즉, 공식 문서만 보면 `CHEST_UPPER` 같은 값은 라이브러리 자체의 public slug가 아니라, 우리 앱 쪽 세부 코드 또는 호환용 alias로 보는 것이 맞다.

## 3. 이번 결정

### 3-1. 저장/도메인 기준값

도메인 기준 근육 코드는 아래 `camelCase` enum으로 본다.

- `chestUpper`
- `chestMid`
- `chestLower`
- `backUpper`
- `backLats`
- `backLower`
- `shoulderFront`
- `shoulderSide`
- `shoulderRear`
- `armBiceps`
- `armTriceps`
- `armForearms`
- `coreAbs`
- `coreObliques`
- `legGlutes`
- `legHamstrings`
- `legQuads`
- `legCalves`
- `legAdductors`
- `legAbductors`
- `neck`

### 3-2. 외부 라이브러리 raw 값

`react-body-highlighter`나 추후 다른 body map UI가 넘기는 값은 `rawBodySelections`로 취급한다.

이 값은:

- 저장 모델의 source of truth가 아니다
- DB 표준 컬럼/enum 이름이 아니다
- AI prompt 입력의 최종 기준도 아니다

### 3-3. 현재 AI 입력값

현재 `routine_engine`은 거친 타깃 문자열 목록을 받는 구조이므로, 최종적으로는 아래 계층을 거친다.

`rawBodySelections -> canonicalMuscleRegions -> aiTargetMuscles`

## 4. 정규화 단계

### 단계 1. Frontend raw 입력 수집

예:

```json
{
  "rawBodySelections": ["chest", "front-deltoids", "triceps"]
}
```

또는 현재 호환 코드 예:

```json
{
  "rawBodySelections": ["CHEST_UPPER", "SHOULDER_FRONT", "ARM_TRICEPS"]
}
```

### 단계 2. Backend / AI Adapter 정규화

예:

```json
{
  "canonicalMuscleRegions": [
    "chestUpper",
    "chestMid",
    "chestLower",
    "shoulderFront",
    "armTriceps"
  ]
}
```

### 단계 3. 현재 AI 요청값으로 축약

예:

```json
{
  "targetMuscles": ["chest", "frontDelts", "triceps"]
}
```

## 6. 구현 원칙

1. 프론트는 라이브러리 raw 값을 임의로 DB 기준값처럼 고정하지 않는다.
2. 백엔드 또는 AI adapter가 normalization 책임을 가진다.
3. `primaryMuscles` 같은 catalog / response 필드는 사용자 친화적인 표현을 유지할 수 있지만, 내부 매핑 기준은 이 문서를 따른다.
4. raw 값은 debugging 또는 snapshot 보존용으로만 남기고, 저장/추천 로직의 기준값은 canonical 값으로 본다.
5. 새 body map 라이브러리로 교체돼도 canonical enum은 되도록 유지한다.

## 7. 이번 스프린트에서 꼭 반영할 것

1. `POST /api/routines/generate` 또는 그 직전 adapter에서 body selection normalization 단계를 명시한다.
2. mock / fixture / 계약 문서가 `coarse display label`과 `canonical region code`를 혼동하지 않도록 한다.
3. 프론트가 `react-body-highlighter` raw slug 또는 현재 세부 코드 둘 중 무엇을 보내든, adapter에서 수용 가능하도록 한다.
4. 현재 repo에 `react-body-highlighter` 의존성이 보이지 않으면, 실제 프론트 브랜치와 `package.json` 동기화 여부를 점검한다.
